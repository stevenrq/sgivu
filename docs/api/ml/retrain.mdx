## Descripción General

La API de Reentrenamiento de Modelo permite a usuarios autorizados disparar el reentrenamiento del modelo de predicción de demanda con datos de transacciones actualizados. Este endpoint obtiene los datos más recientes de compras y ventas, entrena un nuevo modelo y lo despliega para predicciones.

## URL Base

```
https://your-domain.com/v1/ml
```

---

## Reentrenar Modelo

```http
POST /v1/ml/retrain
```

Dispara una operación de reentrenamiento del modelo utilizando datos de transacciones actualizados.

**Autenticación:** Requerida (JWT o clave de servicio interno)

**Autorización:** Requiere el permiso `ml:retrain` (configurable vía `PERMISSIONS_RETRAIN`).

<Warning>
  El reentrenamiento del modelo es una operación intensiva en recursos que puede tardar varios minutos. El gateway mantiene una ruta dedicada `/v1/ml/retrain` con timeout de 1800 s (30 min) y un circuit breaker propio (`mlRetrainCircuitBreaker`). Evita disparar múltiples trabajos concurrentes.
</Warning>

### Cuerpo de la Solicitud

<ParamField body="start_date" type="string" format="date">
  Fecha de inicio para los datos de entrenamiento (ISO 8601: YYYY-MM-DD). Opcional — por defecto usa los datos más antiguos disponibles.
</ParamField>

<ParamField body="end_date" type="string" format="date">
  Fecha de fin para los datos de entrenamiento (ISO 8601: YYYY-MM-DD). Opcional — por defecto usa los datos más recientes.
</ParamField>

```json
{
  "start_date": "2024-01-01",
  "end_date": "2026-03-01"
}
```

<Note>
  Si se omiten tanto `start_date` como `end_date`, el modelo se entrenará con todos los datos históricos disponibles.
</Note>

### Respuesta

<ResponseField name="version" type="string">
  Identificador de versión del modelo recién entrenado
</ResponseField>

<ResponseField name="metrics" type="object">
  Métricas de rendimiento del modelo sobre el conjunto de prueba
</ResponseField>

<ResponseField name="metrics.mae" type="number">
  Error Absoluto Medio (Mean Absolute Error)
</ResponseField>

<ResponseField name="metrics.rmse" type="number">
  Raíz del Error Cuadrático Medio (Root Mean Square Error)
</ResponseField>

<ResponseField name="metrics.mape" type="number">
  Error Porcentual Absoluto Medio (Mean Absolute Percentage Error)
</ResponseField>

<ResponseField name="metrics.wape" type="number">
  Weighted Absolute Percentage Error - Error porcentual ponderado por el volumen real (más estable que MAPE en meses con ventas cercanas a cero). Es la métrica usada por defecto para la selección del mejor modelo.
</ResponseField>

<ResponseField name="metrics.r2" type="number">
  R-cuadrado (coeficiente de determinación)
</ResponseField>

<ResponseField name="metrics.residual_std" type="number">
  Desviación estándar de los residuos sobre el conjunto de prueba. Se utiliza para construir intervalos de confianza.
</ResponseField>

<ResponseField name="metrics.baselines" type="object" nullable>
  Métricas de baselines naive (`naive_lag1_rmse`, etc.) para cuantificar el valor agregado por el modelo entrenado frente a heurísticas simples.

<ResponseField name="trained_at" type="string">

</ResponseField>

<ResponseField name="samples" type="object">

</ResponseField>

<ResponseField name="samples.test" type="integer">

</ResponseField>

<ResponseField name="samples.total" type="integer">
  Número total de muestras
</ResponseField>

### Códigos de Estado

- `200 OK` — Modelo reentrenado exitosamente
- `400 Bad Request` — Rango de fechas o parámetros inválidos
- `401 Unauthorized` — Autenticación faltante o inválida
- `403 Forbidden` — Permisos insuficientes
- `500 Internal Server Error` — Falló el entrenamiento

### Ejemplo

```bash
# Reentrenar con todos los datos disponibles
curl -X POST https://your-domain.com/v1/ml/retrain \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{}'

# Reentrenar con un rango de fechas específico
curl -X POST https://your-domain.com/v1/ml/retrain \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "start_date": "2024-01-01",
    "end_date": "2026-03-01"
  }'
```

## Proceso de Reentrenamiento

El pipeline de reentrenamiento sigue estos pasos:

### 1. Extracción de Datos

- Obtiene transacciones de compras y ventas del servicio `sgivu-purchase-sale`
- Filtra por rango de fechas si se especifica
- Agrega ventas por segmento de vehículo (marca, modelo, línea) y mes

### 2. Ingeniería de Características

- Crea características de series temporales (mes, año, estacionalidad)
- Calcula estadísticas a nivel de segmento
- Genera características de rezago y agregados móviles

### 3. Entrenamiento del Modelo

- Divide los datos en train/test respetando el orden temporal (primeros ~80% de meses para train, últimos ~20% para test)
- Evalúa dos candidatos: Random Forest (`n_estimators=300`) y XGBoost (`n_estimators=500`)
- Aplica búsqueda de hiperparámetros con `RandomizedSearchCV` (`cv_n_iter=10`, `cv_n_jobs=-1`) usando `TimeSeriesSplit(n_splits=3)` para validación temporal
- Selecciona el mejor modelo por **score compuesto ponderado** (modo `weighted` por defecto): `(1 - mape_weight) * RMSE + mape_weight * WAPE`, con `mape_weight=0.4`. Configurable vía `MODEL_SELECTION_METRIC` (`weighted` | `rmse` | `mape`)
- Refita el mejor candidato sobre el dataset completo antes de persistirlo

### 4. Evaluación del Modelo

- Calcula métricas de rendimiento (MAE, RMSE, MAPE, WAPE, R², `residual_std`)
- Computa baselines naive (p.ej. `naive_lag1_rmse`) para validar que el modelo aporta mejora real
- Almacena los `horizon_residuals` por paso para calibrar intervalos de confianza por horizonte

### 5. Despliegue del Modelo

- Persiste los artefactos del modelo usando `joblib`
- Opcionalmente guarda en PostgreSQL
- Actualiza los metadatos del modelo
- Lo activa como el modelo de producción actual

## Versionado de Modelos

A cada modelo entrenado se le asigna un identificador de versión:

**Formato:** `YYYYMMDDHHMMSS` (UTC, sin separadores)

**Ejemplo:** `20260306143522`

- Basado en el timestamp UTC de finalización del entrenamiento
- Las versiones se ordenan cronológicamente de forma natural

### Gestión de Versiones

- Los nuevos modelos se convierten automáticamente en el modelo activo para predicciones
- Las versiones anteriores del modelo se conservan para rollback si es necesario
- Los artefactos del modelo pueden almacenarse tanto en el sistema de archivos como en la base de datos

## Requisitos de Datos de Entrenamiento

### Volumen Mínimo de Datos

Para obtener modelos confiables, asegúrese de contar con:

- **Rango temporal:** Al menos 6 meses de datos históricos por segmento (configurable via `MIN_HISTORY_MONTHS`)
- **Cobertura de segmentos:** Múltiples segmentos de vehículos con transacciones suficientes
- **Tamaño de muestra:** Mínimo 100 transacciones por segmento recomendado

### Calidad de Datos

- Los valores nulos o faltantes se manejan automáticamente
- Los valores atípicos se detectan y pueden ser excluidos
- Los segmentos con datos insuficientes se omiten

<Warning>
  Entrenar con datos insuficientes puede producir bajo rendimiento del modelo. La API devolverá un error si el dataset es demasiado pequeño.
</Warning>

## Métricas de Rendimiento

### Error Absoluto Medio (MAE)

Diferencia absoluta promedio entre las ventas predichas y las reales.

- **Unidades:** Igual que la variable objetivo (número de vehículos)
- **Menor es mejor**
- **Ejemplo:** Un MAE de 1.32 significa que las predicciones se desvían en ~1-2 vehículos en promedio

### Raíz del Error Cuadrático Medio (RMSE)

Raíz cuadrada del promedio de los errores al cuadrado. Penaliza más los errores grandes.

- **Unidades:** Igual que la variable objetivo
- **Menor es mejor**
- **Rango típico:** Ligeramente mayor que el MAE

### Error Porcentual Absoluto Medio (MAPE)

Error porcentual promedio relativo a los valores reales.

- **Unidades:** Porcentaje (0-1)
- **Menor es mejor**
- **Ejemplo:** Un MAPE de 0.14 significa un 14% de error promedio

### Coeficiente de Determinación (R²)

Proporción de varianza explicada por el modelo.

- **Rango:** 0-1 (puede ser negativo para modelos deficientes)
- **Mayor es mejor**
- **Ejemplo:** Un R² de 0.87 significa que el modelo explica el 87% de la varianza

### Interpretación de Métricas

**Buen modelo:**
- MAE < 2.0
- RMSE < 3.0
- MAPE < 0.20 (20%)
- R² > 0.75

**Modelo excelente:**
- MAE < 1.0
- RMSE < 1.5
- MAPE < 0.10 (10%)
- R² > 0.90

## Buenas Prácticas

### Frecuencia de Reentrenamiento

**Frecuencia recomendada:**

- **Semanal:** Para concesionarios con alto volumen de transacciones frecuentes
- **Mensual:** Para operaciones de volumen moderado
- **Trimestral:** Para mercados de bajo volumen o estables

### Cuándo Reentrenar

Activa reentrenamiento cuando:

- Haya nuevos datos de transacciones disponibles
- El rendimiento del modelo se degrade (las predicciones se vuelven menos precisas)
- Ocurran cambios significativos en el mercado (estacionalidad, cambios económicos)
- Se introduzcan nuevos modelos de vehículos

### Monitoreo del Rendimiento del Modelo

Monitorea estos indicadores:

- Precisión de predicciones con datos recientes
- Análisis de residuos (real vs. predicho)
- Tendencias de métricas a lo largo del tiempo
- Cobertura (porcentaje de segmentos con predicciones)

## Programación de Reentrenamiento

El servicio incluye un cron interno habilitado por defecto (`RETRAIN_CRON=0 3 1 * *`, día 1 de cada mes a las 3 AM UTC). Puede modificarse o deshabilitarse vía variable de entorno; tambien puede dispararse externamente con cron jobs o tareas programadas:

### Ejemplo: Cron Job Semanal

```bash
# Reentrenar cada lunes a las 2 AM
0 2 * * 1 curl -X POST https://your-domain.com/v1/ml/retrain \
  -H "X-Internal-Service-Key: $INTERNAL_KEY" \
  -H "Content-Type: application/json" \
  -d '{}'
```

### Ejemplo: Tarea Programada en Python

```python
import requests
import schedule
import time

def retrain_model():
    response = requests.post(
        "https://your-domain.com/v1/ml/retrain",
        headers={
            "X-Internal-Service-Key": "YOUR_KEY",
            "Content-Type": "application/json"
        },
        json={}
    )
    print(f"Estado del reentrenamiento: {response.status_code}")
    print(response.json())

# Programar reentrenamiento semanal
schedule.every().monday.at("02:00").do(retrain_model)

while True:
    schedule.run_pending()
    time.sleep(3600)
```

## Almacenamiento y Persistencia

### Artefactos del Modelo

Los modelos entrenados se persisten usando `joblib`:

- **Ubicación:** Configurada mediante la variable de entorno `MODEL_DIR` (por defecto `models/`)
- **Formato:** Pipeline scikit-learn (preprocesador + estimador) serializado con `joblib.dump`
- **Nomenclatura:** `{MODEL_NAME}_{version}.joblib` (por defecto `demand_forecaster_YYYYMMDDHHMMSS.joblib`)
- **Puntero `latest`**: archivo `latest.json` mantiene la versión activa para acceso rápido

### Persistencia en Base de Datos

Opcionalmente, los metadatos y artefactos del modelo pueden almacenarse en PostgreSQL:

- **Tabla:** `ml_model_artifacts`
- **Campos:** version, metrics, trained_at, artifact_data (binario)

### Snapshots de Features

Las features de entrenamiento pueden guardarse para reproducibilidad:

- **Tabla:** `ml_training_features`
- **Propósito:** Auditoría, depuración, experimentos de reentrenamiento

## Rollback and Model Management

If a newly trained model performs poorly:

1. **Manual Rollback:** Replace current model file with previous version
2. **Database Rollback:** Update active model pointer in database
3. **Verify:** Test predictions with `/predict` endpoint

<Note>
  Future versions will include an API endpoint for automated model rollback.
</Note>
