El Servicio de Compra-Venta (`sgivu-purchase-sale`) orquesta las transacciones de compra y venta de vehículos coordinando datos de los servicios de Usuario, Cliente y Vehículo.

## Características Principales

- Orquestación de transacciones entre múltiples servicios
- Gestión de registros de compras y ventas
- Integración con los servicios de Usuario, Cliente y Vehículo
- Migración de base de datos con Flyway
- API RESTful con documentación OpenAPI
- Autenticación interna entre servicios

## Configuración Base

### Configuración del Servidor

```yaml
server:
  port: ${PORT:8084}
```

### Configuración de JPA

```yaml
spring:
  jpa:
    open-in-view: false
```

<Info>
Deshabilitar open-in-view garantiza límites de transacción adecuados en este servicio de orquestación.
</Info>

### Migración con Flyway

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    baseline-on-migrate: ${FLYWAY_BASELINE_ON_MIGRATE:false}
    baseline-version: 0
    validate-on-migrate: true
```

## Integración de Servicios

### Registro en Eureka

```yaml
eureka:

  client:
```

### Mapa de Descubrimiento de Servicios

Este servicio se integra con **todos los servicios de negocio**:

```yaml
services:
  map:
    sgivu-auth:
      name: sgivu-auth
      url: ${SGIVU_AUTH_URL:http://sgivu-auth:9000}
    sgivu-client:
      name: sgivu-client
      url: ${SGIVU_CLIENT_URL:http://sgivu-client:8082}
    sgivu-user:
      name: sgivu-user
      url: ${SGIVU_USER_URL:http://sgivu-user:8081}
    sgivu-vehicle:
      name: sgivu-vehicle
      url: ${SGIVU_VEHICLE_URL:http://sgivu-vehicle:8083}
```

<Accordion title="Dependencias de Servicios">
- **sgivu-auth**: Validación de tokens y autorización
- **sgivu-client**: Información de clientes para transacciones
- **sgivu-user**: Información de usuarios para el registro de transacciones
- **sgivu-vehicle**: Detalles de vehículos para registros de compra/venta
</Accordion>

<Info>
Este es el único servicio que se comunica con todos los demás servicios de negocio, lo que lo convierte en una verdadera capa de orquestación.
</Info>

### Autenticación Interna entre Servicios

```yaml
service:
  internal:
    secret-key: "${SERVICE_INTERNAL_SECRET_KEY}"
```

El servicio utiliza este secreto para autenticar las llamadas API hacia los servicios de Usuario, Cliente y Vehículo.

## Orquestación de Transacciones

El Servicio de Compra-Venta coordina transacciones complejas:

### Ejemplo de Flujo de Compra

1. Recibir solicitud de compra vía API
2. Validar usuario (llamada al Servicio de Usuario)
3. Validar cliente (llamada al Servicio de Cliente)
4. Validar disponibilidad del vehículo (llamada al Servicio de Vehículo)
5. Crear registro de transacción en la base de datos local
6. Actualizar estado del vehículo (llamada al Servicio de Vehículo)
7. Retornar confirmación de la transacción

### Ejemplo de Flujo de Venta

1. Recibir solicitud de venta vía API
2. Validar vendedor (llamada al Servicio de Usuario)
3. Validar cliente comprador (llamada al Servicio de Cliente)
4. Validar propiedad del vehículo (llamada al Servicio de Vehículo)
5. Crear registro de venta en la base de datos local
6. Transferir propiedad del vehículo (llamada al Servicio de Vehículo)
7. Generar documentación de la venta

<Note>
Considere implementar el patrón Saga para la gestión de transacciones distribuidas con acciones compensatorias.
</Note>

## Endpoints de API

Endpoints típicos del servicio de compra-venta:

```
GET    /api/purchase-sales              - List transactions
GET    /api/purchase-sales/{id}         - Get transaction by ID
POST   /api/purchase-sales/purchase     - Create purchase
POST   /api/purchase-sales/sale         - Create sale
PUT    /api/purchase-sales/{id}         - Update transaction
DELETE /api/purchase-sales/{id}         - Cancel transaction
GET    /api/purchase-sales/client/{id}  - Get client transactions
GET    /api/purchase-sales/vehicle/{id} - Get vehicle history
```

<Info>
Todos los endpoints requieren autenticación a través del gateway.
</Info>

## Modelo de datos de transacción

Una entidad de transacción típica incluye:

- ID y tipo de transacción (compra/venta)
- ID del usuario (quien procesó la transacción)
- ID del cliente
- ID del vehículo
- Monto y moneda de la transacción
- Fecha y estado de la transacción
- Detalles de pago
- Notas y metadatos

## Manejo de errores

El servicio de orquestación debe manejar fallos de forma controlada:

### Patrón Circuit Breaker

Se recomienda implementar circuit breakers para llamadas entre servicios:

```yaml
resilience4j:
  circuitbreaker:
    instances:
      userService:
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10000
      clientService:
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10000
      vehicleService:
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10000
```

### Estrategia de reintentos

```yaml
resilience4j:
  retry:
    instances:
      userService:
        max-attempts: 3
        wait-duration: 1000
```

<Warning>
Implementa siempre timeout y lógica de reintentos al invocar servicios downstream para evitar fallos en cascada.
</Warning>

```java
HttpHeaders headers = new HttpHeaders();
headers.set("X-Internal-Secret", serviceInternalSecretKey);
```

## Consideraciones de rendimiento

### Caché

Cachea datos accedidos frecuentemente desde otros servicios:

```yaml
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=500,expireAfterWrite=10m
```

### Llamadas paralelas a servicios

Cuando se requieran datos de múltiples servicios, haz llamadas en paralelo:

```java
CompletableFuture<User> userFuture = userService.getUserAsync(userId);
CompletableFuture<Client> clientFuture = clientService.getClientAsync(clientId);
CompletableFuture<Vehicle> vehicleFuture = vehicleService.getVehicleAsync(vehicleId);

CompletableFuture.allOf(userFuture, clientFuture, vehicleFuture).join();
```

## Seguridad

### Autenticación Service-to-Service

Todas las llamadas internas de API deben incluir el secreto compartido.

### Autorización de transacciones

Verifica los permisos del usuario antes de permitir la creación de transacciones:

- Validar rol de usuario (admin, ventas, etc.)
- Validar límites de transacción
- Auditar todas las operaciones

## Monitoreo

### Métricas clave

Monitorea estas métricas en el servicio de orquestación:

- Tasas de éxito/fallo de transacciones
- Latencias de llamadas a servicios (User, Client, Vehicle)
- Rendimiento de consultas a base de datos
- Tasa de errores por servicio
- Estado de los circuit breakers

### Alertas

Configura alertas para:

- High failure rates
- Service unavailability
- Slow transaction processing
- Circuit breaker opens
