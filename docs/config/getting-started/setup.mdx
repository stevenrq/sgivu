<Info>
  El repositorio de configuración de SGIVU es un proyecto de infraestructura basado en Spring Cloud Config que centraliza la gestión de configuración para todos los microservicios de SGIVU en diferentes entornos.
</Info>

## Requisitos Previos

Antes de configurar el Config Server, asegúrate de tener instalado lo siguiente:

<CardGroup cols={2}>
  <Card title="Java 17+" icon="java">
    Necesario para ejecutar aplicaciones Spring Boot
  </Card>
  <Card title="Docker y Docker Compose" icon="docker">
    Para el despliegue contenerizado de los servicios
  </Card>
  <Card title="Git" icon="git">
    Para clonar y versionar el repositorio de configuración
  </Card>
  <Card title="PostgreSQL" icon="database">
    Base de datos para los microservicios (Auth, User, Client, etc.)
  </Card>
</CardGroup>

## Pasos de Instalación

<Steps>
  <Step title="Clonar el Repositorio de Configuración">
    Clona este repositorio en tu máquina local:

    ```bash
    git clone <repository-url> sgivu-config-repo
    cd sgivu-config-repo
    ```

    Este repositorio contiene todos los archivos de configuración YAML organizados por servicio y entorno:
    - `{servicio}.yml` - Configuración base
    - `{servicio}-dev.yml` - Sobrescrituras de desarrollo
    - `{servicio}-prod.yml` - Sobrescrituras de producción
  </Step>

  <Step title="Configurar las Variables de Entorno Requeridas">
    Los archivos de configuración usan placeholders de variables de entorno que deben resolverse. Crea un archivo `.env` con las siguientes variables:

    <CodeGroup>
    ```bash Development (.env)
    # Database Configuration - Auth Service
    DEV_AUTH_DB_HOST=localhost
    DEV_AUTH_DB_PORT=5432
    DEV_AUTH_DB_NAME=sgivu_auth
    DEV_AUTH_DB_USERNAME=postgres
    DEV_AUTH_DB_PASSWORD=your_password

    # Database Configuration - User Service
    DEV_USER_DB_HOST=localhost
    DEV_USER_DB_PORT=5432
    DEV_USER_DB_NAME=sgivu_user
    DEV_USER_DB_USERNAME=postgres
    DEV_USER_DB_PASSWORD=your_password

    # Database Configuration - Client Service
    DEV_CLIENT_DB_HOST=localhost
    DEV_CLIENT_DB_PORT=5432
    DEV_CLIENT_DB_NAME=sgivu_client
    DEV_CLIENT_DB_USERNAME=postgres
    DEV_CLIENT_DB_PASSWORD=your_password

    # Redis Configuration (for Gateway sessions)
    REDIS_HOST=localhost
    REDIS_PORT=6379
    REDIS_PASSWORD=your_redis_password

    # Service URLs
    EUREKA_URL=http://localhost:8761/eureka
    SGIVU_AUTH_URL=http://localhost:9000
    SGIVU_GATEWAY_URL=http://localhost:8080
    DEV_ANGULAR_APP_URL=http://localhost:4200

    # Security Configuration
    SGIVU_GATEWAY_SECRET=your_gateway_client_secret
    SERVICE_INTERNAL_SECRET_KEY=your_internal_service_key

    # JWT Keystore Configuration
    JWT_KEYSTORE_LOCATION=classpath:keystore.jks
    JWT_KEYSTORE_PASSWORD=your_keystore_password
    JWT_KEY_ALIAS=sgivu-jwt
    JWT_KEY_PASSWORD=your_key_password

    # Flyway
    FLYWAY_BASELINE_ON_MIGRATE=false
    ```

    ```bash Production (.env)
    # Usar valores de producción para todas las variables
    # Las bases de datos deben apuntar a instancias de producción
    # Los secretos deben gestionarse con AWS Secrets Manager o equivalente
    
    DEV_AUTH_DB_HOST=prod-db-host
    # ... (similar structure with production values)
    ```
    </CodeGroup>

    <Warning>
      **Nunca subas el archivo `.env` al control de versiones.** Agrégalo a `.gitignore` inmediatamente. Usa gestión de secretos específica por entorno para despliegues en producción.
    </Warning>
  </Step>

  <Step title="Configurar el Spring Cloud Config Server">
    El Config Server (típicamente `sgivu-config`) debe configurarse para usar este repositorio como fuente de configuración.

    En el archivo `application.yml` del Config Server, agrega:

    ```yaml application.yml
    spring:
      application:
        name: sgivu-config
      cloud:
        config:
          server:
            git:
              uri: file:///path/to/sgivu-config-repo
              # Or for remote repository:
              # uri: https://github.com/your-org/sgivu-config-repo
              # username: ${GIT_USERNAME}
              # password: ${GIT_TOKEN}
              default-label: main
              clone-on-start: true
              search-paths: .
      profiles:
        active: native  # Use 'native' for local file system

    server:
      port: 8888

    eureka:
      client:
        service-url:
          defaultZone: ${EUREKA_URL:http://localhost:8761/eureka}
    ```

    <Note>
      Para despliegues con Docker, usa el enfoque de URI de Git. Para desarrollo local, puedes usar el perfil `native` con una ruta del sistema de archivos.
    </Note>
  </Step>

  <Step title="Iniciar el Config Server">
    Inicia el Spring Cloud Config Server usando Docker Compose:

    ```bash
    # Desde el directorio sgivu-docker-compose
    docker compose up -d sgivu-config
    ```

    O ejecútalo localmente con Maven/Gradle:

    ```bash
    # Si ejecutas el Config Server de forma standalone
    ./mvnw spring-boot:run
    # or
    ./gradlew bootRun
    ```

    El Config Server se iniciará en el puerto `8888` por defecto.
  </Step>

  <Step title="Verificar los Endpoints de Configuración">
    Comprueba que el Config Server está sirviendo las configuraciones correctamente:

    ```bash
    # Test auth service configuration (dev profile)
    curl http://localhost:8888/sgivu-auth/dev | jq

    # Test gateway configuration (dev profile)
    curl http://localhost:8888/sgivu-gateway/dev | jq

    # Test user service configuration (default profile)
    curl http://localhost:8888/sgivu-user/default | jq
    ```

    Deberías ver una salida JSON con las propiedades de configuración resueltas. La respuesta incluirá:
    - `name` - Nombre del servicio
    - `profiles` - Perfiles activos
    - `propertySources` - Configuración combinada de los archivos base + específicos del perfil

    <CodeGroup>
    ```json Expected Response
    {
      "name": "sgivu-auth",
      "profiles": ["dev"],
      "label": null,
      "version": "abc123",
      "state": null,
      "propertySources": [
        {
          "name": "file:///path/to/sgivu-config-repo/sgivu-auth-dev.yml",
          "source": {
            "spring.datasource.url": "jdbc:postgresql://localhost:5432/sgivu_auth",
            "spring.jpa.hibernate.ddl-auto": "validate",
            // ... more properties
          }
        }
      ]
    }
    ```
    </CodeGroup>
  </Step>

  <Step title="Configurar los Microservicios para Usar el Config Server">
    Cada microservicio necesita configurarse para obtener su configuración desde el Config Server.

    Agrega lo siguiente al archivo `bootstrap.yml` o `application.yml` de cada servicio:

    ```yaml bootstrap.yml
    spring:
      application:
        name: sgivu-auth  # Debe coincidir con el nombre del archivo de configuración
      cloud:
        config:
          uri: http://localhost:8888
          # For Docker: http://sgivu-config:8888
          fail-fast: true
          retry:
            max-attempts: 6
            initial-interval: 1000
            max-interval: 2000
            multiplier: 1.1
      profiles:
        active: dev  # or prod
    ```

    <Info>
      El valor de `spring.application.name` debe coincidir con el nombre base del archivo de configuración (por ejemplo, `sgivu-auth` coincide con `sgivu-auth.yml` y `sgivu-auth-dev.yml`).
    </Info>
  </Step>
</Steps>

## Estructura de Archivos de Configuración

El repositorio sigue las convenciones de nombres de Spring Cloud Config:

```plaintext
sgivu-config-repo/
├── sgivu-auth.yml           # Base configuration for Auth service
├── sgivu-auth-dev.yml       # Development overrides
├── sgivu-auth-prod.yml      # Production overrides
├── sgivu-gateway.yml        # Base configuration for Gateway
├── sgivu-gateway-dev.yml
├── sgivu-gateway-prod.yml
├── sgivu-user.yml
├── sgivu-user-dev.yml
├── sgivu-user-prod.yml
├── sgivu-client.yml
├── sgivu-client-dev.yml
├── sgivu-client-prod.yml
├── sgivu-vehicle.yml
├── sgivu-vehicle-dev.yml
├── sgivu-vehicle-prod.yml
├── sgivu-purchase-sale.yml
├── sgivu-purchase-sale-dev.yml
├── sgivu-purchase-sale-prod.yml
├── sgivu-discovery.yml
├── sgivu-discovery-dev.yml
└── sgivu-discovery-prod.yml
```

<Note>
  Spring Cloud Config combina las configuraciones en orden: archivo base → archivo específico del perfil. Los valores del perfil sobrescriben los valores base.
</Note>

## Patrones Clave de Configuración

### Placeholders de Variables de Entorno

Todos los datos sensibles usan la sintaxis de placeholders de propiedades de Spring:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DEV_AUTH_DB_HOST:host.docker.internal}:${DEV_AUTH_DB_PORT:5432}/${DEV_AUTH_DB_NAME}
    username: ${DEV_AUTH_DB_USERNAME}
    password: ${DEV_AUTH_DB_PASSWORD}
```

Formato: `${VAR_NAME:valor_por_defecto}` o `${VAR_NAME}` (falla si no está definida)

### Integración con Service Discovery

Todos los servicios se registran en Eureka:

```yaml
eureka:
  instance:
    instance-id: ${spring.cloud.client.hostname}:${spring.application.name}:${random.value}
  client:
    service-url:
      defaultZone: ${EUREKA_URL:http://sgivu-discovery:8761/eureka}
```

### Endpoints de Gestión

Los perfiles de desarrollo exponen todos los endpoints de actuator para depuración:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "*"  # dev only
  endpoint:
    health:
      show-details: always
```

## Mejores Prácticas de Seguridad

<Warning>
  **Directrices Críticas de Seguridad:**
  - Nunca subas secretos reales a los archivos YAML
  - Usa variables de entorno o gestores de secretos (AWS Secrets Manager, Vault, etc.)
  - Mantén el repositorio privado con controles de acceso apropiados
  - Rota los secretos periódicamente
  - Usa secretos diferentes para cada entorno
</Warning>

## Solución de Problemas

### El Config Server No Inicia

- Verifica que la ruta o URL del repositorio Git sea correcta
- Comprueba la conectividad de red si usas Git remoto
- Asegúrate de que las variables de entorno requeridas estén configuradas
- Revisa los logs: `docker logs sgivu-config`

### Los Servicios No Pueden Conectarse al Config Server

- Verifica que el Config Server esté corriendo: `curl http://localhost:8888/actuator/health`
- Comprueba `spring.cloud.config.uri` en la configuración del servicio
- Asegúrate de que haya conectividad de red (red Docker para contenedores)
- Verifica que el nombre del servicio coincida con el nombre del archivo de configuración

### La Configuración No Se Actualiza

- Verifica que los cambios estén commiteados y pusheados (para el backend Git)
- Revisa los logs del Config Server en busca de eventos de refresco
- Los servicios pueden necesitar un reinicio o una llamada a `/actuator/refresh`
- Consulta [Desarrollo Local](/config/getting-started/local-development) para la configuración de recarga en caliente
