## Descripción General

La gestión adecuada de secretos es fundamental para mantener la seguridad de tu infraestructura. El repositorio de configuración de SGIVU utiliza una combinación de placeholders de variables de entorno y gestores de secretos externos para mantener los datos sensibles fuera del control de versiones.

<Warning>
  **Nunca confirmes secretos reales en archivos YAML.** Todos los valores sensibles deben externalizarse mediante variables de entorno o sistemas de gestión de secretos.
</Warning>

## Placeholders de Variables de Entorno

### Sintaxis

Spring Cloud Config soporta la sintaxis de placeholders `${VAR_NAME:default}` donde:
- `VAR_NAME` es la variable de entorno a leer
- `default` es un valor de respaldo opcional (puede omitirse para secretos obligatorios)

### Ejemplos del Repositorio

#### Credenciales de Base de Datos

En `sgivu-auth-dev.yml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DEV_AUTH_DB_HOST:host.docker.internal}:${DEV_AUTH_DB_PORT:5432}/${DEV_AUTH_DB_NAME}
    username: ${DEV_AUTH_DB_USERNAME}
    password: ${DEV_AUTH_DB_PASSWORD}
```

<Info>
  Observa cómo `DEV_AUTH_DB_HOST` y `DEV_AUTH_DB_PORT` tienen valores por defecto para desarrollo local con Docker, mientras que `DEV_AUTH_DB_NAME`, `DEV_AUTH_DB_USERNAME` y `DEV_AUTH_DB_PASSWORD` son obligatorios y no tienen valores por defecto.
</Info>

#### Secretos de Cliente OAuth2

En `sgivu-gateway.yml`:

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          sgivu-gateway:
            client-secret: ${SGIVU_GATEWAY_SECRET}
```

#### Claves Internas de Servicio

En `sgivu-auth.yml`:

```yaml
gateway-client:
  url: ${SGIVU_GATEWAY_URL:http://sgivu-gateway:8080}
  secret: ${SGIVU_GATEWAY_SECRET}

service:
  internal:
    secret-key: ${SERVICE_INTERNAL_SECRET_KEY}
```

#### Configuración del Keystore JWT

En `sgivu-auth.yml`:

```yaml
sgivu:
  jwt:
    keystore:
      location: ${JWT_KEYSTORE_LOCATION}
      password: ${JWT_KEYSTORE_PASSWORD}
    key:
      alias: ${JWT_KEY_ALIAS}
      password: ${JWT_KEY_PASSWORD}
```

<Tip>
  Usa nombres de variables descriptivos que indiquen el entorno y el propósito, como `DEV_AUTH_DB_PASSWORD` en lugar de solo `DB_PASSWORD`.
</Tip>

## Desarrollo Local con Archivos .env

Para desarrollo local, crea un archivo `.env` en tu directorio `sgivu-docker-compose`:

```bash
# Database credentials
DEV_AUTH_DB_NAME=sgivu_auth_dev
DEV_AUTH_DB_USERNAME=sgivu_dev
DEV_AUTH_DB_PASSWORD=dev_password_123

# Service secrets
SGIVU_GATEWAY_SECRET=gateway_secret_dev
SERVICE_INTERNAL_SECRET_KEY=internal_key_dev

# Redis
REDIS_PASSWORD=redis_dev_password

# JWT Configuration
JWT_KEYSTORE_LOCATION=classpath:keystore.jks
JWT_KEYSTORE_PASSWORD=keystore_password
JWT_KEY_ALIAS=sgivu-jwt
JWT_KEY_PASSWORD=key_password
```

<Warning>
  Agrega `.env` a tu `.gitignore` para evitar confirmar secretos locales accidentalmente.
</Warning>

## Separación de Secretos por Entorno

Usa nombres de variables específicos por entorno para prevenir contaminación entre entornos:

<Steps>
  <Step title="Desarrollo">
    Usa el prefijo `DEV_` para todos los secretos de desarrollo
    
    ```yaml
    username: ${DEV_AUTH_DB_USERNAME}
    password: ${DEV_AUTH_DB_PASSWORD}
    ```
  </Step>
  
  <Step title="Producción">
    Usa el prefijo `PROD_` para todos los secretos de producción
    
    ```yaml
    username: ${PROD_AUTH_DB_USERNAME}
    password: ${PROD_AUTH_DB_PASSWORD}
    ```
  </Step>
  
  <Step title="Validar la Separación">
    Asegúrate de que ningún servicio pueda usar accidentalmente credenciales de producción en desarrollo manteniendo nombres de variables distintos
  </Step>
</Steps>

## Gestores de Secretos Externos

Para entornos de producción, considera integrar sistemas de gestión de secretos externos:

  </Card>

### Patrón de Integración

1. Almacena los secretos en tu gestor de secretos
2. Configura el Config Server para leer desde el gestor de secretos
3. Usa la misma sintaxis de placeholders en los archivos YAML
4. Deja que Spring Cloud Config resuelva los secretos en tiempo de ejecución

## Errores Comunes a Evitar

<Warning>
  ### Secretos Hardcodeados
  
  **Nunca hagas esto:**
  ```yaml
  spring:
    datasource:
      password: MyP@ssw0rd123  # ❌ Hardcoded secret
  ```
  
  **Siempre haz esto:**
  ```yaml
  spring:
    datasource:
      password: ${PROD_AUTH_DB_PASSWORD}  # ✅ Externalized
  ```
</Warning>

<Warning>
  ### Confirmar Archivos .env
  
  Los archivos `.env` locales **nunca** deben confirmarse en el control de versiones. Siempre agrégalos al `.gitignore`.
</Warning>

<Warning>
  ### Usar Secretos de Producción en Desarrollo
  
  Nunca reutilices credenciales de producción en entornos de desarrollo. Siempre mantén secretos separados con prefijos específicos por entorno.
</Warning>

<Warning>
  ### Valores por Defecto Débiles
  
  Si debes proveer valores por defecto, asegúrate de que solo sean adecuados para desarrollo local:
  
  ```yaml
  # ❌ Bad - weak default that might be used in production
  password: ${DB_PASSWORD:password}
  
  # ✅ Good - no default, forces explicit configuration
  password: ${PROD_DB_PASSWORD}
  ```
</Warning>

## Lista de Verificación de Seguridad

Antes de confirmar cambios de configuración:

- [ ] Todos los valores sensibles usan placeholders `${VAR_NAME}`
- [ ] No hay contraseñas, tokens ni claves API hardcodeadas
- [ ] Las variables específicas por entorno usan prefijos apropiados (`DEV_`, `PROD_`)
- [ ] Los valores por defecto (si existen) son seguros solo para desarrollo local
- [ ] Los archivos `.env` están en `.gitignore`
- [ ] El repositorio tiene controles de acceso apropiados
- [ ] Los secretos están documentados en el gestor de contraseñas del equipo o sistema de secretos

## Actualización de Secretos

Cuando los secretos cambian, los servicios necesitan obtener los nuevos valores:

### Usando Docker Compose

```bash
# Restart the Config Server to reload secrets
docker compose restart sgivu-config

# Restart affected microservices
docker compose restart sgivu-auth sgivu-gateway
```

### Usando Spring Boot Actuator

Para servicios que soportan actualización dinámica:

```bash
curl -X POST http://localhost:9000/actuator/refresh
```

<Info>
  El endpoint `/actuator/refresh` debe estar habilitado en la configuración del servicio y generalmente requiere autenticación en producción.
</Info>

## Resumen de Buenas Prácticas

1. **Nunca confirmes secretos** — Usa variables de entorno para todos los datos sensibles
2. **Usa nombres descriptivos** — Incluye el entorno y el propósito en los nombres de las variables
3. **Provee valores por defecto seguros** — Solo para desarrollo local, nunca para producción
4. **Separa por entorno** — Usa prefijos `DEV_`, `PROD_` para prevenir accidentes
5. **Usa gestores de secretos** — Integra con AWS Secrets Manager, Vault, etc. para producción
6. **Documenta los secretos** — Mantén un inventario seguro de qué secretos existen y dónde
7. **Rota periódicamente** — Establece un proceso para rotar credenciales de forma regular
8. **Audita el acceso** — Monitorea quién accede al repositorio de configuración y a los sistemas de secretos
