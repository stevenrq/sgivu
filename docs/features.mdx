# Funcionalidades de SGIVU

SGIVU proporciona un conjunto integral de funcionalidades diseñadas para la gestión moderna de inventario de vehículos. Esta página cubre todas las capacidades principales en autenticación, dominios de negocio, aprendizaje automático y servicios de plataforma.

## Autenticación y Autorización

### OAuth 2.1 / OpenID Connect

<CardGroup cols={2}>
  <Card title="Autenticación Moderna" icon="fingerprint">
    Implementación completa de OAuth 2.1 y OpenID Connect con mejores prácticas de la industria
  </Card>
  <Card title="Tokens JWT" icon="key">
    Tokens de acceso JWT firmados con algoritmo RS256 usando keystore JKS
  </Card>
  <Card title="Flujo PKCE" icon="shield-check">
    Authorization Code con PKCE para aplicaciones de página única seguras
  </Card>
  <Card title="Tokens de Actualización" icon="arrows-rotate">
    Renovación automática de tokens para una experiencia de usuario fluida
  </Card>
</CardGroup>

**Características Principales:**

- **Emisión de Tokens**: Tokens de acceso (JWT) y tokens de actualización
- **Descubrimiento OIDC**: Endpoint estándar `.well-known/openid-configuration`
- **Endpoint JWKS**: Claves públicas en `/oauth2/jwks` para verificación de tokens
- **Claims Personalizados**: Incluye `rolesAndPermissions`, `username`, `isAdmin`
- **Gestión de Sesiones**: Spring Session JDBC para sesiones OAuth2 persistentes
- **Múltiples Clientes**: Soporte para aplicaciones web, móviles y herramientas de prueba

**Endpoints Soportados:**

```bash
# OIDC Discovery
GET /.well-known/openid-configuration

# Authorization
GET /oauth2/authorize

# Token endpoint
POST /oauth2/token

# JWKS
GET /oauth2/jwks

# Login page
GET /login
```

### Patrón Backend for Frontend (BFF)

<Info>
El API Gateway implementa el patrón BFF, gestionando tokens OAuth2 del lado del servidor para mayor seguridad.
</Info>

**Beneficios del BFF:**

<Steps>
  <Step title="Seguridad de Tokens">
    Los tokens de acceso y actualización se almacenan del lado del servidor en Redis, nunca expuestos al navegador
  </Step>
  <Step title="Cookie de Sesión">
    Cookie de sesión simple con atributos `HttpOnly`, `SameSite=Lax`
  </Step>
  <Step title="Relay Automático de Tokens">
    El Gateway incluye automáticamente tokens Bearer al hacer proxy hacia los servicios backend
  </Step>
  <Step title="Escalado Horizontal">
    Las sesiones respaldadas por Redis permiten instancias de gateway sin estado
  </Step>
</Steps>

**API de Sesión:**

```bash
# Get current session info
GET /auth/session

Response:
{
  "subject": "user-uuid",
  "username": "john.doe",
  "roles": ["USER", "ADMIN"],
  "isAdmin": true
}
```

### Sistema de Permisos Granular

**Modelo de Permisos:**

SGIVU utiliza permisos detallados mapeados a claims JWT:

```
user:create, user:read, user:update, user:delete
person:create, person:read, person:update, person:delete
role:create, role:read, role:update, role:delete
permission:read
vehicle:create, vehicle:read, vehicle:update, vehicle:delete
client:create, client:read, client:update, client:delete
contract:create, contract:read, contract:update, contract:delete
ml:predict, ml:retrain
```

**Control de Acceso Basado en Roles:**

- Los roles agrupan múltiples permisos
- Los usuarios pueden tener múltiples roles
- Los permisos se verifican vía anotaciones `@PreAuthorize`
- El JWT incluye el array `rolesAndPermissions` para validación eficiente

**Ejemplo de Autorización:**

```java
@PreAuthorize("hasAuthority('vehicle:create')")
public VehicleResponse createVehicle(VehicleRequest request) {
    // Only users with vehicle:create permission can execute
}
```

### Autenticación entre Servicios

**Clave Interna de Servicio:**

Para comunicación entre servicios sin contexto de usuario:

```bash
GET /internal/users/username/{username}
X-Internal-Service-Key: {shared-secret}
```

<Warning>
La clave interna de servicio debe mantenerse secreta y almacenarse en variables de entorno o en un gestor de secretos. Nunca debe incluirse en Git.
</Warning>

### Gestión de Imágenes con S3

**Flujo de Carga con URL Prefirmada:**

<Steps>
  <Step title="Solicitar URL de Carga">
    El frontend solicita una URL prefirmada al backend
    ```bash
    POST /v1/vehicles/{vehicleId}/images/upload-url
    Content-Type: application/json
    
    {
      "filename": "front-view.jpg",
      "contentType": "image/jpeg"
    }
    ```
  </Step>
  
  <Step title="Recibir URL Prefirmada">
    El backend genera una URL prefirmada válida por 5 minutos
    ```json
    {
      "uploadUrl": "https://s3.amazonaws.com/...",
      "key": "vehicles/uuid/images/front-view.jpg",
      "expiresIn": 300
    }
    ```
  </Step>
  
  <Step title="Cargar Directamente a S3">
    El cliente sube la imagen directamente a S3 usando la URL prefirmada
    ```bash
    PUT {uploadUrl}
    Content-Type: image/jpeg
    
    {binary image data}
    ```
  </Step>
  
  <Step title="Confirmar Carga">
    Tras la carga exitosa, confirmar para registrar los metadatos
    ```bash
    POST /v1/vehicles/{vehicleId}/images/confirm-upload
    
    {
      "key": "vehicles/uuid/images/front-view.jpg",
      "filename": "front-view.jpg",
      "contentType": "image/jpeg",
      "size": 245678
    }
    ```
  </Step>
</Steps>

**Características de Imágenes:**

- **Formatos Soportados**: JPEG, PNG, WebP
- **Imagen Principal**: Primera imagen o designada manualmente
- **Ordenamiento Automático**: Si se elimina la principal, se promueve la siguiente
- **Configuración CORS**: Bucket S3 configurado para orígenes permitidos
- **Almacenamiento de Metadatos**: Información de imágenes almacenada en PostgreSQL
- **Descargas con URL Prefirmada**: URLs de descarga seguras con tiempo limitado

**APIs de Gestión de Imágenes:**

```bash
# List vehicle images
GET /v1/vehicles/{id}/images

# Get presigned download URL
GET /v1/vehicles/{id}/images/{imageId}/download-url

# Set primary image
PUT /v1/vehicles/{id}/images/{imageId}/primary

# Delete image
DELETE /v1/vehicles/{id}/images/{imageId}
```

<Note>
La carga directa a S3 reduce la carga del backend y mejora el rendimiento de las subidas. El backend valida y realiza seguimiento de los metadatos después de la carga.
</Note>

### Circuit Breaking y Resiliencia

**Integración con Resilience4j:**

<CardGroup cols={2}>
  <Card title="Circuit Breaker" icon="circle-exclamation">
    Previene fallos en cascada con circuit breaking automático
  </Card>
  <Card title="Rutas de Fallback" icon="route">
    Degradación elegante con respuestas de fallback
  </Card>
</CardGroup>

**Configuración del Circuit Breaker:**

```yaml
resilience4j:
  circuitbreaker:
    instances:
      vehicleService:
        failure-rate-threshold: 50
        slow-call-rate-threshold: 50
        slow-call-duration-threshold: 2s
        wait-duration-in-open-state: 10s
        sliding-window-size: 10
        minimum-number-of-calls: 5
```

**Estados del Circuit Breaker:**

1. **CLOSED**: Operación normal, las solicitudes fluyen normalmente
2. **OPEN**: Demasiados fallos, las solicitudes se rechazan inmediatamente
3. **HALF_OPEN**: Probando si el servicio se ha recuperado

**Manejo de Fallback:**

```bash
# When circuit opens, Gateway returns fallback
GET /v1/vehicles/search

# Fallback response:
{
  "error": "Service temporarily unavailable",
  "fallback": true,
  "message": "Vehicle service is experiencing issues. Please try again later.",
  "statusCode": 503
}
```

## Database Features

### Flyway Migrations

**Version-Controlled Schema:**

```bash
src/main/resources/db/migration/
├── V1__initial_schema.sql
├── V2__add_vehicle_images.sql
├── V3__add_contract_status.sql
└── R__seed_data.sql (repeatable)
```

**Migration Features:**

- Automatic execution on startup
- Version tracking in `flyway_schema_history` table
- Rollback support (via scripts)
- Repeatable migrations for seed data
- Checksum validation

### Database Per Service

**Isolation:**

Each microservice has its own database:

- **Data Isolation**: Service owns its data schema
- **Independent Scaling**: Scale databases independently
- **Technology Freedom**: Different services can use different DB technologies
- **Fault Isolation**: Database failure affects only one service

### PostgreSQL Features

- **Indexes**: Optimized queries with strategic indexes
- **Constraints**: Foreign keys, unique constraints, check constraints
- **JSON Support**: JSONB columns for flexible data
- **Full-Text Search**: Text search capabilities
- **Audit Columns**: `created_at`, `updated_at` timestamps

## Docker & Deployment

### Containerization

<Info>
All services are containerized with multi-stage Docker builds for optimal image size.
</Info>

**Docker Features:**

- Multi-stage builds (build → runtime)
- Minimal base images (Alpine, Distroless)
- Non-root users for security
- Health checks in Dockerfile
- Build scripts for automation

**Example Multi-Stage Build:**

```dockerfile
# Stage 1: Build
FROM eclipse-temurin:21-jdk-alpine AS build
WORKDIR /workspace
COPY mvnw .
COPY .mvn .mvn
COPY pom.xml .
RUN ./mvnw dependency:go-offline
COPY src src
RUN ./mvnw package -DskipTests

# Stage 2: Runtime
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=build /workspace/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```
