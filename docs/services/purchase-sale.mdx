## Descripción General

`sgivu-purchase-sale` gestiona los contratos de compra y venta de vehículos. Proporciona creación, búsqueda avanzada, generación de informes (PDF/XLSX/CSV) y gestión del ciclo de vida de los contratos. El servicio se integra con `sgivu-user`, `sgivu-client` y `sgivu-vehicle` para el enriquecimiento de datos, y con `sgivu-auth` para la autorización.

## Tecnologías

- **Java**: 25
- **Spring Boot**: 4.0.1
- **Spring Security**: Resource Server (validación JWT)
- **Spring Data JPA**: Persistencia en PostgreSQL
- **Flyway**: Migraciones de base de datos
- **Spring Cloud Config Client**: Configuración centralizada
- **Spring Cloud Eureka Client**: Registro de servicios
- **SpringDoc OpenAPI**: Documentación Swagger UI
- **MapStruct**: Mapeo de DTOs
- **Lombok**: Generación de código
- **OpenPDF**: Generación de informes PDF
- **Apache POI**: Generación de informes Excel (XLSX)
- **Spring Cache + Caffeine**: Caché L1 in-memory
- **Spring Data Redis**: Caché L2 distribuida (TTL ~60 s en `dashboard-summary`)

## Tipos de Contrato

### Contrato de Compra

**Cuándo**: La empresa adquiere un vehículo de un proveedor/particular

**Campos Clave**:
- `contractType`: `PURCHASE`
- `clientId`: ID del vendedor/proveedor (sgivu-client)
- `vehicleId`: Vehículo adquirido (sgivu-vehicle)
- `userId`: Empleado responsable (sgivu-user)
- `purchasePrice`: Monto pagado para adquirir el vehículo
- `paymentMethod`: cualquiera de los 9 valores soportados
- `paymentLimitations`, `paymentTerms`, `observations`: opcionales

### Contrato de Venta

**Cuándo**: La empresa vende un vehículo a un cliente

**Campos Clave**:
- `contractType`: `SALE`
- `clientId`: ID del comprador/cliente
- `vehicleId`: Vehículo en venta
- `userId`: Representante de ventas
- `salePrice`: Monto que paga el cliente
- `paymentMethod`, `paymentLimitations`, `paymentTerms`, `observations`

## Flujo de Estados del Contrato

`ContractStatus` tiene exactamente cuatro valores reales:

<Steps>
  <Step title="PENDING">
    Estado inicial al crear el contrato. Transiciones permitidas: `ACTIVE`, `COMPLETED`, `CANCELED`.
  </Step>

  <Step title="ACTIVE">
    Contrato en curso (pago en cuotas, financiamiento, entrega parcial). Transiciones permitidas: `COMPLETED`, `CANCELED`.
  </Step>

  <Step title="COMPLETED">
    Contrato ejecutado y transacción cerrada. **Estado terminal**.
  </Step>

  <Step title="CANCELED">
    Contrato cancelado antes de la finalización. **Estado terminal**.
  </Step>
</Steps>

<Note>
Cada cambio de estado se persiste automáticamente en la tabla `contract_status_history` (V2 migration), capturando estado anterior, nuevo, usuario que realizó el cambio, timestamp y motivo opcional.
</Note>

## Seguridad

### Autenticación

**Validación de Tokens JWT**:
- Tokens emitidos por `sgivu-auth`
- `JwtAuthenticationConverter` mapea el claim `rolesAndPermissions` a authorities
- El Resource Server valida la firma mediante el endpoint JWKS

### Autorización

**Permisos Requeridos**:
- `purchase_sale:create` - Crear contratos
- `purchase_sale:read` - Ver contratos, generar informes (PDF/XLSX/CSV) y consultar `dashboard-summary`
- `purchase_sale:update` - Actualizar contratos
- `purchase_sale:delete` - Eliminar contratos (solo perfil dev)

### Comunicación Interna entre Servicios

**Header X-Internal-Service-Key**:
- Permite llamadas entre servicios sin JWT
- Utilizado para endpoints internos
- Debe mantenerse en secreto

<Warning>
La clave interna de servicio omite la autenticación. Asegúrese de que:
- La clave esté almacenada en un gestor de secretos
- Solo los servicios autorizados tengan acceso
- La clave se rote periódicamente
- El Gateway bloquee los endpoints `/internal/*`
</Warning>

## Esquema de Base de Datos

### Migraciones Flyway

**Ubicación**: `src/main/resources/db/migration`

**V1__initial_schema.sql** crea:

**Tabla purchase_sales**:
- `id`: Clave primaria
- `contract_type`: ENUM (`PURCHASE`, `SALE`)
- `contract_status`: ENUM (`PENDING`, `ACTIVE`, `COMPLETED`, `CANCELED`)
- `payment_method`: ENUM con 9 valores (`CASH`, `BANK_TRANSFER`, `BANK_DEPOSIT`, `CASHIERS_CHECK`, `MIXED`, `FINANCING`, `DIGITAL_WALLET`, `TRADE_IN`, `INSTALLMENT_PAYMENT`)
- `client_id`: Referencia al cliente (sgivu-client)
- `user_id`: Referencia al usuario responsable (sgivu-user)
- `vehicle_id`: Referencia al vehículo (sgivu-vehicle)
- `purchase_price`, `sale_price`: precios numéricos (uno u otro según `contract_type`)
- `payment_limitations`, `payment_terms`, `observations`: campos de texto opcional
- `created_at`, `updated_at`: Marcas de tiempo de auditoría

**V2__add_contract_status_history.sql** añade:

**Tabla contract_status_history** (audita transiciones de estado):
- `id`: Clave primaria
- `purchase_sale_id`: FK al contrato
- `previous_status`, `new_status`: ENUM `ContractStatus` (previous puede ser NULL en la creación)
- `changed_by`: ID del usuario que realizó el cambio
- `changed_at`: timestamp del cambio
- `reason`: motivo opcional del cambio

**R__demo_data.sql** (migración repetible) carga ~12 000 contratos sintéticos con un modelo de demanda estacional para pruebas y entrenamiento ML.

## Integración de Servicios

### Enriquecimiento de Datos

Las respuestas de contratos incluyen datos enriquecidos de servicios relacionados:

**Información del Cliente** (desde `sgivu-client`):
- Persona: Nombre completo, número de documento
- Empresa: Razón social, CUIT/RUC

**Información del Usuario** (desde `sgivu-user`):
- Nombre de usuario, nombre completo
- Información de rol

**Información del Vehículo** (desde `sgivu-vehicle`):
- VIN, marca, modelo, año
- Estado actual
- Imagen principal

### Patrón de Comunicación

```java
// Example: Enriching contract with client data
@Service
public class ContractEnrichmentService {

    @LoadBalanced
    private final WebClient webClient;

    public PurchaseSaleDTO enrichContract(PurchaseSale contract) {
        // Fetch client data
        ClientDTO client = webClient.get()
            .uri("http://sgivu-client/v1/clients/{id}", contract.getClientId())
            .header("X-Internal-Service-Key", internalKey)
            .retrieve()
            .bodyToMono(ClientDTO.class)
            .block();

        // Similar for user and vehicle...

        return mapToDTO(contract, client, user, vehicle);
    }
}
```

## Solución de Problemas

<AccordionGroup>
  <Accordion title="Errores de autorización 401/403">
    **Síntomas**: Las solicitudes a la API retornan no autorizado o prohibido

    **Soluciones**:
    - Verificar que el token Bearer contenga los permisos requeridos
    - Comprobar que el emisor del token coincida con `sgivu-auth`
    - Asegurarse de que el usuario tenga las asignaciones de rol apropiadas
    - Validar que el token no haya expirado
  </Accordion>

  <Accordion title="Fallos de integración">
    **Síntomas**: No se pueden obtener datos de cliente/usuario/vehículo

    **Soluciones**:
    - Verificar que `sgivu-config` y `sgivu-discovery` estén accesibles
    - Comprobar que los servicios relacionados estén registrados en Eureka
    - Asegurarse de que la clave interna de servicio sea correcta
    - Revisar la conectividad de red entre servicios
    - Verificar si el circuit breaker está en estado abierto
  </Accordion>

  <Accordion title="Transición de estado inválida">
    **Síntomas**: La actualización de estado se rechaza con error de validación

    **Soluciones**:
    - Revisar las transiciones permitidas en el flujo de trabajo
    - No se pueden actualizar contratos en estados terminales (`COMPLETED`, `CANCELED`)
    - Verificar el estado actual antes de intentar la transición
    - Comprobar que el usuario tenga permiso de aprobación para ciertas transiciones
  </Accordion>

  <Accordion title="Problemas en la generación de informes">
    **Síntomas**: La generación de PDF/Excel falla o se agota el tiempo de espera

    **Soluciones**:
    - Verificar las dependencias de OpenPDF y Apache POI
    - Revisar los límites de memoria (los informes pueden consumir mucha memoria)
    - Limitar el tamaño del conjunto de resultados para informes grandes
    - Verificar los permisos de escritura en el sistema de archivos
    - Revisar los logs en busca de errores específicos de la librería
    - Considerar la generación asíncrona de informes para grandes volúmenes de datos
  </Accordion>

  <Accordion title="Violaciones de restricciones de base de datos">
    **Síntomas**: La creación del contrato falla con error de restricción FK

    **Soluciones**:
    - Verificar que client_id exista en el servicio de clientes
    - Verificar que user_id exista en el servicio de usuarios
    - Verificar que vehicle_id exista en el servicio de vehículos
    - Comprobar que el vehículo no esté ya en un contrato activo
    - Asegurarse de que las entidades referenciadas no estén eliminadas lógicamente
  </Accordion>
</AccordionGroup>
